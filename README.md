# Vai Lang

> **Experimental project** — the API and syntax may change. Not yet production-ready.

Vai is a language that manages prompts and obligations to generate clear instructions for LLMs — without over-engineering and without forcing them to read thousands of lines of code.

---

## The Problem

AI-assisted coding tools (Claude Code, Cursor, Copilot) all work the same way: they read your project, build a plan, generate code. But they have structural limitations that get worse as the project grows.

### 1. Wasted tokens on every request

Every time you ask for a change, the tool re-reads from scratch the parts of the project it considers relevant. There's no memory of the previous context. On a large project, the cost in tokens (and time) becomes enormous — even for a trivial change.

### 2. Throwaway plans

The tool builds an implementation plan, you approve it, code gets generated. The plan is discarded. On the next request, everything starts from zero: no trace of architectural decisions, no accumulated context. The result is that code quality degrades progressively — the model loses the big picture.

### 3. Uncontrolled code exposure

Tools like Claude Code, if not properly configured, can read any source file in the project — including files with sensitive implementations, keys, proprietary logic. You have no granular control over what the model sees.

### 4. One expensive model for everything

Today the same large (and expensive) model is used both for planning the architecture and for writing the body of a single function. It's wasteful: you don't need a 200B parameter model to implement a `for` loop.

---

## The Solution

Vai addresses these problems with a different approach:

**Persistent plans, not throwaway markdown.** `.vai` and `.plan` files stay in the project. They contain specs, constraints, and implementation tasks in a structured format that the Vai compiler can read, validate, and transform into optimized prompts.

**Two-tier system.** Vai splits the work into two phases with two different models:
- The **Architect** (large, capable model) receives the spec and produces the skeleton: imports, types, function signatures with compilable stubs.
- The **Executor** (small, fast model — even local) receives one function at a time with its dependencies and writes the implementation.

**Explicit control.** You decide exactly which files the model sees (`target`, `reference`) and which symbols are available (`[use]`). No indiscriminate reading of the project.

**Smart caching.** The `vai.lock` file tracks the hash of every plan and implementation. If nothing has changed, Vai skips execution. You don't pay twice for the same work.

## How It Works — The Cycle

```
  .vai / .plan file
         |
         v
  +--- Vai Compiler -------+
  |  Reader -> Lexer        |
  |  Parser -> Composer     |
  +-----------+-------------+
              |
              v
     Optimized prompt
              |
              v
  +--- 5-Step Pipeline -----+
  |                          |
  |  1. Architect (LLM big)  |    The large model receives the spec
  |         |                |    and produces the file skeleton
  |         v                |
  |  2. Diff (tree-sitter)   |    The skeleton is applied
  |         |                |    to the target file in memory
  |         v                |
  |  3. Save                 |    The impls are saved
  |         |                |    back into the original .vai file
  |         v                |
  |  4. Executor (LLM small) |    Each function is implemented
  |         |                |    in parallel by the small model
  |         v                |
  |  5. Debug (loop)         |    Compile check — if errors,
  |                          |    the model fixes them (max 3 attempts)
  +--------------------------+
              |
              v
     Generated source file
```

### Step 1 — Architect

The large model (e.g. Claude Sonnet) receives the full spec: what to build, the constraints, the current state of the target files. It returns:
- All necessary imports
- Type and struct declarations
- Function signatures with stub bodies (`todo!()` in Rust, `panic("todo")` in Go, `pass` in Python)
- One implementation task per function (2-5 lines of instructions)

The architect sees the complete picture but **does not write implementations**.

### Step 2 — Diff

Vai uses tree-sitter to apply the skeleton to the target files in memory. Imports are inserted, declarations added or modified with byte-range precision. No approximate text manipulation.

### Step 3 — Save

The implementation instructions generated by the architect are saved directly into the original `.vai` file as `impl` blocks. The plan becomes persistent — you can review it, modify it, re-execute it.

### Step 4 — Executor

Each `impl` is sent **in parallel** to the small model (e.g. Claude Haiku, or a local model). The executor receives:
- The function's stub code
- The architect's instructions
- The dependency signatures (`[use]`)

It doesn't see the rest of the project. It doesn't know what Vai is. It receives a clean prompt and writes one function.

### Step 5 — Debug

Vai compiles the result with the target language's compiler. If there are errors, it analyzes them with language-specific diagnostic parsers and asks the model to fix them. It repeats up to 3 times (configurable).

## Practical Example

### The `.plan` file

```vai
plan todo {
    target "src/todo.c"

    spec {
        Build a simple in-memory todo application.
        Use a fixed-size array of 100 Todo structs.
        Provide a CLI menu loop: add, list, complete, delete, quit.
    }

    constraint {
        Use strncpy, never strcpy.
        Check array bounds before every insertion.
        C99 only — no extensions.
    }

    impl add_todo {
        Add a new todo to the array.
        Prompt the user for the title using fgets.
        Check bounds before inserting.
        [use Todo]
        [use MAX_TODOS]
    }

    impl list_todos {
        Print all todos with index, status, and title.
        [use Todo]
    }

    impl main {
        REPL loop: print menu, read choice, dispatch.
        Use switch statement for menu options.
        [use add_todo]
        [use list_todos]
        [use complete_todo]
        [use delete_todo]
    }
}
```

### What happens

**`vai build todo.plan`** — compiles the file and shows the generated prompt. No LLM calls. Useful to verify that the syntax is correct and the prompt makes sense.

**`vai gen`** — runs the full pipeline:
1. The architect receives the spec and constraints, produces the skeleton for `src/todo.c` with structs, constants, and signatures
2. The skeleton is applied to the file
3. The impls are saved into the `.plan`
4. Each function is implemented in parallel by the small model
5. `gcc -Wall -Werror` verifies the result, errors are fixed automatically

**`vai gen skeleton`** — generates the skeleton: writes stubs and assigns implementation tasks to the target file. Useful to verify the structure before generating code.

**`vai gen plan`** — reshapes the spec for the skeleton. Useful to refine the plan before generating stubs.

**`vai gen code`** — runs only the executor. Requires the skeleton to be already saved. Useful to regenerate implementations without re-running the architect.

**`vai prompt "add error handling"`** — natural language interface. Describe what you want in plain text: the LLM reads your existing plans, modifies the specs, and then runs the full pipeline automatically.

## Vai Syntax

### Declarations

| Keyword      | Purpose                                              | Scope          |
|-------------|------------------------------------------------------|----------------|
| `prompt`    | Reusable prompt text block                           | Top-level      |
| `plan`      | Structured scope with specs, constraints, impls      | Top-level      |
| `spec`      | Detailed specification                               | Inside `plan`  |
| `constraint`| Rules the model must follow                          | Inside `plan`  |
| `impl`      | Implementation task for a named symbol               | Inside `plan`  |
| `target`    | Output file path                                     | Inside `plan`  |
| `reference` | Symbol source for `[use]` resolution (not in output) | Anywhere       |
| `inject`    | Include a prompt's content                           | Top-level      |
| `inspect`   | Raw plan content extraction (eval-only)              | Top-level      |

### Body Directives

| Directive              | Purpose                                            |
|-----------------------|----------------------------------------------------|
| `[use X]`             | Dependency on symbol X (signature only)            |
| `[use X+code]`        | Include full source code of X                      |
| `[use X+doc]`         | Include documentation of X                         |
| `[inject X]`          | Inline a prompt's content                          |
| `[target "path"]`     | Reference a target file                            |
| `[reference "path"]`  | Load symbols for `[use]` resolution                |
| `[reference plan]`    | Import another plan's targets as reference symbols |
| `[match field]`       | Conditional block based on config value            |
| `[case "value"]`      | Case branch inside a match block                   |

### How `[use]` works

When you write `[use Todo]` in an `impl` block, Vai:
1. Looks up the symbol `Todo` in the plan's `target` and `reference` files
2. Uses tree-sitter to extract its signature (e.g. `struct Todo { char title[256]; int completed; }`)
3. Inserts the signature into the prompt sent to the executor

The executor sees exactly the dependencies it needs — nothing more. This reduces tokens and prevents context dilution.

## Configuration (`vai.toml`)

```toml
[lib]
name = "myproject"
prompts = "plans"          # directory containing .vai and .plan files

[planner]
provider = "anthropic"     # anthropic, openai, gemini, or custom
model = "claude-sonnet-4-6"
max_tokens = 8000

[executor]
provider = "anthropic"     # can be a different provider from the planner
model = "claude-haiku-4-5"
max_tokens = 4000

[debug]
max_attempts = 3

[debug.languages.c]
compile_check = "gcc -Wall -Werror -o /tmp/output {target}"
tools = []

[debug.languages.rust]
compile_check = "cargo check --message-format=json"
format = "json"
tools = ["cargo fmt"]
```

The planner and executor can use different providers. You can use Claude for the architect and a local model for the executor. Or Gemini for the architect and OpenAI for the executor. Any combination.

## CLI Commands

| Command                  | What it does                                               |
|-------------------------|------------------------------------------------------------|
| `vai init <name>`       | Scaffold a vai.toml                                        |
| `vai build [file.vai]`  | Compile and show the prompt (no LLM)                       |
| `vai build --eval <expr>` | Evaluate a vai expression                                |
| `vai tree`              | Show the package tree                                      |
| `vai gen`               | Full pipeline: architect + diff + save + executor + debug  |
| `vai gen skeleton`      | Generate skeleton: write stubs and assign implementation tasks |
| `vai gen plan`          | Reshape the spec for the skeleton                          |
| `vai gen code`          | Executor only: implement functions (skeleton required)     |
| `vai prompt <message>`  | Natural language interface: create or modify plans via LLM, then run full pipeline |
| `vai gen --name <plan>` | Run only a specific plan                                   |
| `vai gen --force`       | Ignore cache (vai.lock) and re-execute everything          |
| `vai gen --json`        | Output in JSON format                                      |

## Why Vai

| Problem                          | Without Vai                            | With Vai                                 |
|----------------------------------|----------------------------------------|------------------------------------------|
| Context lost between requests    | Start from scratch every time          | Plans stay in the project                |
| Token cost                       | Large model for everything             | Large for structure only, small for code |
| Control over exposed files       | The tool reads whatever it wants       | You declare target and reference         |
| Repeated work                    | Regenerates even unchanged parts       | vai.lock skips unchanged parts           |
| Quality on large projects        | Degrades with project size             | Each impl is isolated and focused        |

---

## Supported Languages

Vai uses tree-sitter for symbol extraction and supports:

- **Go**
- **C**
- **Python**
- **Rust**
- **TypeScript**

For each language, Vai can extract function signatures, structs/classes, enums, import zones, and skeletons. The diagnostic parser in the debug step is also specialized per language.
